Yet Another Template System User Guide

YATS is a template system designed as a library with helper functions for Code Igniter 1.5 and up. This library was designed after the CI philosophies—it’s lean and straight forward. After installing YATS you’ll see how easy is to create and edit templates for your projects! 

Here’s some features of YATS:
- Multiple templates support
- Configurable items
- Helper functions
- Modular structure
- Page building with just one line of code
- Resources folder outside the MVC structure
- Permissive license
- Easy to design for and easy for designers

Installing

To get YATS up and running is quite easy, just a few steps, and you’ll be running Code Igniter with a complete template system. See how you should do it.

Drag and Drop

1. Uncompress the downloaded file on your desktop and open the folder
2. Copy the folder application to your CI folder (if you’re on a mac you’ll have to copy the files individually)
3. Copy the assets folder to your CI folder along with the system folder

Configuring

1. Open the file application/config/autoload.php on your favorite text editor
2. Look for $autoload[’libraries’] and set it to array(’layout’);
3. Look for $autoload[’helper’] and set it to array(’html’, ‘url’, ‘layout’);
4. Save and close it

Optionally you can edit or even add settings to YATS. For that just open the file application/config/layout.php and make your changes.

Now what?

Just open a browser and hit your CI project, all should be running flawlessly. 


Using YATS

You can pretty much do any crazy graphic design for your page, and YATS will build it together with CI for your users. Yet, before we dig in something needs to be clarified, and here it is: leave the graphic design in the CSS file and all should be okay. In other words, all data you generate in CI is there on all view files. So create the layout of your dreams that YATS will get the job done for you.

Theme Files

Each theme has its own folder inside the assets folder which by default is [your-ci-project-folder]/assets/[theme]. There you’ll see a folder for each type of common file that we use for making web pages. Just drop your theme files inside its folders and you can easily display it on the views.

Assets folder structure

assets
     L theme_name
                L css
                L images
                L script

View Files

Each theme has its own folder inside the application/views folder, and its files are divided in two groups, commons and everything else. In the commons folder you’ll place files shared by the theme, such as header.php, footer.php, and your custom layout elements files as well.

The other groups of files are the views you create for your controllers. In order to keep things organized you should create a folder for each controller and there a view file for each action this controller does. For example we have a controller called welcome, then we create a folder after it, and inside that folder we create a view file to display the data which the controller generates, in this case home.php.

View folder structure

views
    L theme_name
    |           L common
    |          |       L footer. php
    |          |       L header. php                
    |          |       L hello.php                
    |           L content
    |                   L module_name
    |                               L main.php
    |                               L form.php
    |
    L loader.php

As you can see here the view files are modular. So you’ll never get lost while developing a project that needs a great deal of custom views.


Building Layout Elements

Inside the layout_model.php we will code all common layout elements. For example, if a project has a menu we create a function inside this class, and when the page is built YATS automatically will call this function and display the menu. Following you can see a template for these functions.

function whatever ()
 {     
    $retval['whatever'] = "Hello world!";
    return $this->layout->load->view($this->theme . "whatever", $retval, true);
}

On the first line inside the function the $retval variable is where you’ll put the common data that is going to be sent to the function’s view. On the second line we return the data completely formatted to the YATS library. See that after $this->theme we have ‘whatever’ between double quotes, that’s the view file for this function.

Once you’ve coded the function, open the file config/layout.php and look for this array $config[’layout_elements’], and add it there. The function’s name as a key and its parameters as value, if you have more than one parameter set an array for them. This way YATS will automatically generate and make that element available to all CI’s view system.


Building The Whole Thing

Here we can see how simple is to send a page to the user from any given controller. We have just two functions buildPage, which you can call to build and send a page to the user, and for dumping a view to a variable use dumpPage. Both functions work with a syntax that is very similar to the one used by CI views. Read more about this on following topic, More library tricks.

Now to go for the kill we simply code something like this:

$this->layout->buildPage('welcome/home' ,$data);


More library tricks

YATS library does other things besides building pages for you. It can change configuration items on the fly such as themes and layout elements parameters. Following you can see these functions along with its details.


buildPage()

This is the one responsible for all page building as you could see in the previous example. The first parameter is mandatory; here put the view path you want to display. The second is optional; it's an array holding the contents of the page. The third is also optional; it's the place for you to set the name of an alternative loader.

Prototype:

$this ->layout->buildPage('view-path', $data-array, 'alternative-loader');

dumpPage()

It works very much like builPage(), but its behavior is slightly different. Instead of displaying a page this will dump it in a variable for you. The only difference is the last parameter, it's a boolean value set FALSE by default that determines whether or not the entire page should be dumped.

Prototype:

$this ->layout->dumpPage('view_path', $data_array, 'page-dump-type');

resetTheme()

There are times when a theme must be switched on the fly, and for that you should use this function. It has just one parameter, a valid theme name.

Prototype:

$this ->layout->resetTheme('theme-name');

resetView()

Switching views folders is what this function does. As is with resetTheme(), you have to provide a valid view folder name.

Prototype:

$this ->layout->resetView('view-folder-name');

resetElement()

This resets an item from the layout_elements configuration array. It has two parameters, the first is the function name you want to change, and the other is for its new parameters. Also you can reset multiple elements simultaneously by passing an array having keys as function names and values as new parameters.

Prototype:

$this ->layout->resetElement('function-name', 'new-parameters');


Need a Helper?

YATS has a built in helper, which well helps you :-). It’s a set of functions to make things easier while writing views. Using this help is entirely up to you since all that these functions do is write code for you, code which is there in form of theme properties or layout variables. However, some functions may give you advantages such as the possibility of running custom filtering routines before displaying data. Following you can see these functions along with its details.


display()

Tests and outputs a template variable to display it on a view. The first parameter is a string with the template variable or data to display. The second is an array with the filtering routine calls. Once called it will display the data if the optional filtering ran successfully or it will simply echo null.

Prototype:

display("template-variable", array("library" => "function"));

The filtering is more like a shortcut. For example, we need to display a hello user message, however we wouldn’t need to show it for google’s spider. So following we see an example code where we call the is_browser the CI’s agent library.

Also you could use this function to test whether or not a user is logged therefore leaving for the helper to decide if the content is to be shown.

Example:

display("hello_user", array("agent" => "is_browser"));

dump()

This function works exactly like display(). However, instead of echoing the data it will just return it.
Prototype:

dump("template-variable", array("library" => "function"));

property()

Outputs a template property, which are those config variables, started by ‘app_’ (See more about these variables on the following topic called Creating Application Properties).

Prototype:

property("template-property");

style()

It outputs a css link tag with the specified file. The first parameter is the filename inside the template’s css folder. The second is an array with additional attributes, having the key as the attribute name and the value as value.

Prototype:

style("archive.css", additional-attributes);

Example:

style("main.css", array('media'=>'screen', 'charset'=>'utf-8'));

script()

It outputs a script tag to include javascript files. The only parameter is the filename inside the template’s script folder.

Prototype:

script("archive.js");

Example:

style("init.js");

image()

It outputs an <img> tag. The first parameter is the image’s filename. The second is a ‘alt’ description. The third is an array with additional attributes, having the key as the attribute name and the value as value.

Prototype:

image("image.yyz", alt-attribute, additional-attributes);

Example:

image("movingpictures.jpg" , "Moving Pictures", array('style'=>'border:0;float:right;margin:10px;' ));

favicon()

It outputs a favicon tag having just one parameter, the icon file name. This file must be inside the images folder.

Prototype:

favicon("icon.ico");

Example:

favicon("logo.ico");


Configuring YATS

Customizing YATS is quite easy. We have three groups of settings having the prefixes: ‘layout_’, ‘views_’, and ‘assets_’. Following you'll see descriptions for all these settings. Also there is another group initiated by ‘app_’; you can read more about this on Creating Application Properties.


Library Settings

views_folder: path to the default theme view files
views_commons: path to common elements folder inside view/theme
views_content: path to content elements folder inside view/theme
assets_folder: path to the assets folder
assets_design: path to the default theme assets
assets_shared: path to a shared assets folder
assets_styles: path to css folder inside assets
assets_images: path to images folder inside assets
assets_script: path to javascript folder inside assets
layout_model: common elements model name layout_elements: references all functions in this model so the library can automatically call each one of them.


Layout Elements

References all the functions in this model so the Library can automatically call each one of them. Don’t forget to write them on the layout_model.php or whatever is your custom name for it.

Prototype:

array("function" => "parameter", ...);

Where function is the function name and parameter is a single value or an array of them to send to that function as we’d usually do with a hard coded call.


Creating Application Properties

In this section of the config/layout.php file you can come up with any setting you find necessary for your project. From a site title to anything you need. Just remember that in order to work all properties must have the ‘app_’ prefix.
